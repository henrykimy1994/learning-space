자료 흐름도(DFD) 이해하기

May 18, 2025
8 min read
시스템 분석 및 설계
ENG

자료 흐름도(DFD)는 시스템 내에서 데이터가 어떻게 이동하는지 시각적으로 보여주는 강력한 도구입니다. 프로세스, 데이터 흐름, 데이터 저장소, 터미네이터 등의 구성 요소를 통해 복잡한 시스템을 직관적으로 이해할 수 있습니다.

자료 흐름도(DFD)란 무엇인가?

자료 흐름도(Data Flow Diagram, DFD)는 시스템 내에서 데이터가 어떻게 이동하고 처리되는지를 시각적으로 표현하는 모델링 도구입니다. 이는 시스템 분석가와 설계자가 시스템의 구조와 데이터 흐름을 이해하고 문서화하는 데 사용됩니다.

자료 흐름도의 주요 장점

  • 가시성: 복잡한 시스템을 쉽게 이해할 수 있는 시각적 표현 제공
  • 커뮤니케이션: 기술적/비기술적 이해관계자 간의 효과적인 소통 도구
  • 단순성: 시스템의 논리적 관점에 집중하여 불필요한 구현 세부 사항 제거

자료 흐름도는 다음과 같은 네 가지 핵심 구성 요소로 이루어집니다:

  1. 프로세스(Process) - 데이터를 변환하거나 처리하는 작업
  2. 데이터 흐름(Data Flow) - 한 지점에서 다른 지점으로 이동하는 데이터
  3. 데이터 저장소(Data Store) - 데이터가 저장되는 위치
  4. 터미네이터(Terminator) - 시스템 외부에 있는 데이터 출처나 목적지

1. 프로세스(Process)

프로세스는 자료 흐름도의 가장 중요한 구성 요소 중 하나로, 데이터를 변환하거나 처리하는 작업을 의미합니다. 이는 입력 데이터를 받아 출력 데이터로 변환하는 작업이라고 생각할 수 있습니다.

프로세스의 특징:

  • 데이터에 대한 어떤 작업이 수행되는지 나타냄
  • 동사(verb)로 이름을 짓는 것이 일반적 (예: '고객 정보 검증', '주문 처리')
  • 항상 입력 데이터를 받아 출력 데이터를 생성해야 함
프로세스 표기법 

┌─────────────────┐
│       1        │ ← 프로세스 번호
│                │
│   주문 접수    │ ← 프로세스 이름
│                │
└─────────────────┘

또는 타원/원형으로 표현:

   ╭───────────╮
  │     2      │
  │            │
  │  결제 처리  │
  │            │
   ╰───────────╯

예를 들어, 온라인 쇼핑몰 시스템에서 다음과 같은 프로세스들이 있을 수 있습니다:

  • 주문 접수: 고객의 주문 정보를 받아 처리
  • 재고 확인: 주문된 상품의 재고 상태 확인
  • 결제 처리: 고객의 결제 정보를 처리하고 검증
  • 배송 준비: 배송을 위한 정보 생성 및 처리

2. 데이터 흐름(Data Flow)

데이터 흐름은 한 지점에서 다른 지점으로 이동하는 데이터를 나타냅니다. 이는 프로세스, 데이터 저장소, 외부 개체(터미네이터) 사이의 데이터 이동을 보여줍니다.

데이터 흐름의 특징:

  • 항상 방향성이 있음 (화살표로 표시)
  • 명사(noun)로 이름을 짓는 것이 일반적 (예: '주문 정보', '결제 내역')
  • 단일 데이터 항목 또는 데이터 그룹을 나타낼 수 있음
데이터 흐름 표기법 

프로세스 A         프로세스 B
┌────────┐        ┌────────┐
│        │        │        │
│        │        │        │
└────────┘        └────────┘
     │                 ▲  
     │                 │
     │    주문 정보    │
     │                 │
     └────────────────┘

온라인 쇼핑몰 예시에서 데이터 흐름의 예:

  • 고객 정보: 고객으로부터 '고객 정보 확인' 프로세스로
  • 주문 상세: '주문 접수' 프로세스에서 '재고 확인' 프로세스로
  • 결제 내역: '결제 처리' 프로세스에서 결제 데이터베이스로
  • 배송 주소: 고객 데이터베이스에서 '배송 준비' 프로세스로

3. 데이터 저장소(Data Store)

데이터 저장소는 시스템 내에서 데이터가 보관되는 장소를 나타냅니다. 이는 데이터베이스, 파일, 테이블 등 데이터가 일정 기간 동안 저장되는 모든 유형의 저장소를 포함합니다.

데이터 저장소의 특징:

  • 수동적인 요소로, 데이터를 변환하지 않음
  • 최소한 하나의 입력 데이터 흐름과 하나의 출력 데이터 흐름이 있어야 함
  • 보통 복수형 명사로 이름을 짓는 것이 일반적 (예: '고객들', '주문들')
데이터 저장소 표기법 

┌─────────────────────────────────────┐
│ D1 고객 데이터베이스                │
└─────────────────────────────────────┘

또는 평행선으로 표현:

─────────────────────────────────────
 D2 주문 데이터베이스
─────────────────────────────────────

온라인 쇼핑몰 예시에서 데이터 저장소의 예:

  • D1 고객 데이터베이스: 고객 정보(이름, 연락처, 주소 등) 저장
  • D2 상품 카탈로그: 모든 상품 정보와 재고 현황 저장
  • D3 주문 내역: 처리된 모든 주문의 세부 정보 저장
  • D4 결제 기록: 모든 결제 트랜잭션 세부 정보 저장

4. 터미네이터(Terminator)

터미네이터(또는 외부 개체)는 시스템 경계 외부에 있는 데이터의 출처나 목적지를 나타냅니다. 이는 시스템과 상호작용하지만 시스템의 직접적인 부분이 아닌 사람, 조직, 또는 다른 시스템을 포함합니다.

터미네이터의 특징:

  • 시스템 범위 밖에 있는 개체
  • 시스템과 데이터를 주고받음
  • 시스템에 의해 변경되거나 제어되지 않음
  • 보통 명사로 이름을 짓는 것이 일반적 (예: '고객', '공급업체')
터미네이터 표기법 

┌─────────────────┐
│                 │
│      고객       │
│                 │
└─────────────────┘

온라인 쇼핑몰 예시에서 터미네이터의 예:

  • 고객: 주문을 제출하고 결제하는 사람
  • 공급업체: 상품을 제공하는 회사나 조직
  • 배송 회사: 주문된 상품을 배달하는 외부 서비스
  • 결제 게이트웨이: 결제를 처리하는 외부 시스템

자료 흐름도의 계층 구조

자료 흐름도는 다양한 수준의 상세함을 보여주기 위해 여러 계층으로 구성될 수 있습니다. 이를 통해 시스템의 '확대(zoom-in)' 및 '축소(zoom-out)' 뷰를 제공합니다.

DFD의 계층 레벨:

  1. 레벨 0 (상황도 또는 문맥도): 전체 시스템을 하나의 프로세스로 표현하고 주요 입력, 출력, 외부 개체를 보여줌
  2. 레벨 1: 레벨 0의 주요 프로세스를 더 작은 프로세스로 분해하여, 주요 프로세스와 데이터 저장소 간의 관계를 보여줌
  3. 레벨 2 이상: 레벨 1의 각 프로세스를 더 작은 하위 프로세스로 세분화하여 추가적인 세부 정보 제공
graph TD subgraph "레벨 0 (상황도)" A[고객] -->|주문 정보| B((온라인 쇼핑몰 시스템)) B -->|주문 확인| A C[공급업체] -->|재고 업데이트| B B -->|주문 요청| C D[배송 회사] -->|배송 상태| B B -->|배송 요청| D end

레벨 0 DFD: 전체 시스템을 하나의 프로세스로 표현

graph TD A[고객] -->|고객 정보| P1((1. 고객 등록)) P1 -->|저장| DS1[(D1 고객 DB)] A -->|주문 정보| P2((2. 주문 처리)) DS1 -->|고객 정보 조회| P2 P2 -->|주문 저장| DS2[(D2 주문 DB)] P2 -->|재고 확인| P3((3. 재고 관리)) P3 -->|재고 정보| DS3[(D3 재고 DB)] B[공급업체] -->|재고 업데이트| P3 P2 -->|결제 요청| P4((4. 결제 처리)) P4 -->|결제 정보| DS4[(D4 결제 DB)] P4 -->|결제 확인| A P2 -->|배송 요청| P5((5. 배송 관리)) DS2 -->|주문 정보| P5 P5 -->|배송 요청| C[배송 회사] C -->|배송 상태| P5 P5 -->|배송 상태 업데이트| DS2

레벨 1 DFD: 주요 프로세스와 데이터 저장소 간의 관계를 더 자세히 보여줌

실제 예제: 피자 주문 시스템

피자 주문 시스템을 통해 자료 흐름도의 구성 요소들이 어떻게 상호작용하는지 살펴보겠습니다.

레벨 0 DFD - 피자 주문 시스템

레벨 0 DFD의 주요 구성 요소

  • 터미네이터: 고객, 배달원 - 시스템과 상호작용하는 외부 개체
  • 프로세스: 피자 주문 시스템 - 중앙 처리 단위
  • 데이터 흐름: 주문 요청, 영수증, 배달 정보, 배달 확인 - 데이터의 이동

레벨 1 DFD - 피자 주문 시스템

레벨 1 DFD의 주요 구성 요소

  • 터미네이터: 고객, 배달원 - 외부 개체
  • 프로세스: 주문 접수, 피자 제작, 배달 준비 - 주요 처리 단계
  • 데이터 저장소: 주문 데이터베이스, 고객 정보 데이터베이스 - 데이터 보관 장소
  • 데이터 흐름: 주문 요청, 제작 지시, 완성된 피자, 배달 정보 등 - 정보의 이동

이 예제에서 볼 수 있듯이, 레벨이 깊어질수록 시스템의 세부 사항이 더 명확하게 표현됩니다. 레벨 0은 시스템의 대략적인 개요를 보여주는 반면, 레벨 1은 각 프로세스의 세부 단계와 데이터 저장소를 보여주어 시스템의 내부 작동을 더 잘 이해할 수 있게 해줍니다.

자료 흐름도 작성 지침

효과적인 자료 흐름도를 작성하기 위한 몇 가지 중요한 지침을 살펴보겠습니다.

자료 흐름도 작성 모범 사례

  • 명확한 이름 사용: 모든 구성 요소에 명확하고 설명적인 이름을 사용합니다.
  • 균형 있는 입력/출력: 모든 프로세스는 최소한 하나의 입력과 하나의 출력이 있어야 합니다.
  • 데이터 저장소의 적절한 연결: 데이터 저장소는 직접 연결되지 않으며, 항상 프로세스를 통해 데이터가 이동해야 합니다.
  • 터미네이터 간 직접 연결 금지: 터미네이터끼리 직접 연결되지 않습니다. 데이터는 항상 시스템을 통해 이동해야 합니다.
  • 과도한 복잡성 피하기: 각 다이어그램에 너무 많은 프로세스(일반적으로 7±2개 이내)를 포함하지 않도록 합니다.
  • 적절한 번호 매기기: 프로세스와 데이터 저장소에 일관된 번호 체계를 사용합니다.

전문가 팁

자료 흐름도를 작성할 때는 먼저 레벨 0(상황도)부터 시작하여 전체 시스템의 범위를 정의한 다음, 점진적으로 더 자세한 레벨로 확장하는 것이 좋습니다. 이렇게 하면 큰 그림을 먼저 이해한 후 세부 사항에 집중할 수 있습니다.

DFD vs 다른 모델링 도구

모델링 도구 주요 초점 장점 한계
자료 흐름도 (DFD) 데이터 흐름 및 변환 데이터 중심, 직관적, 여러 레벨의 추상화 시간적 측면이나 제어 흐름을 잘 표현하지 못함
UML 다이어그램 객체 구조 및 상호작용 객체 지향, 다양한 다이어그램 유형 초보자에게 복잡할 수 있음
ER 다이어그램 데이터 구조 및 관계 데이터베이스 설계에 적합 프로세스나 기능을 표현하지 않음
순서도 (Flowchart) 프로세스 흐름 및 결정 알고리즘 표현에 적합 데이터 구조나 저장을 표현하지 않음
BPMN 비즈니스 프로세스 비즈니스 흐름 표현에 최적화 기술적 세부 사항을 표현하지 않음

각각의 모델링 도구는 서로 다른 관점에서 시스템을 바라보며, 특정 상황에 더 적합할 수 있습니다. 자료 흐름도는 특히 데이터가 시스템을 통해 어떻게 흐르고 변환되는지 이해하는 데 뛰어난 도구입니다.
