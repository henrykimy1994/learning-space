Coad-Yourdon 방법론 이해하기

May 17, 2025
10 min read
CIPE
ENG

Coad-Yourdon 방법론이 시각적 표현과 구조화된 프로세스를 통해 복잡한 시스템을 모델링하는 직관적인 접근 방식으로 객체 지향 분석을 어떻게 혁신했는지 알아보세요.

Coad-Yourdon 방법론이란 무엇인가?

Coad-Yourdon 방법론은 1990년대 초 피터 코드와 에드워드 유돈이 개발한 객체 지향 분석 및 설계 접근 방식입니다. 이 방법론은 객체 지향적 관점에서 복잡한 시스템을 이해하고 모델링하기 위한 체계적인 프레임워크를 제공합니다.

핵심 철학

  • 단순성: 복잡한 시스템을 쉽게 이해할 수 있는 구조로 표현
  • 안정성: 시스템이 변경되어도 견고하게 유지되는 모델 생성
  • 재사용성: 개발 효율성 향상을 위한 객체 및 컴포넌트의 재사용 촉진

코드-유돈 방법론이 특히 가치 있는 이유는 객체 동작을 모델링하기 위해 엔티티-관계(E-R) 다이어그램을 사용한다는 점입니다. 이 방법론은 다음과 같은 여섯 가지 핵심 프로세스로 구성됩니다:

  1. 객체 식별 - 시스템에서 중요한 엔티티 찾기
  2. 구조 식별 - 관계 및 계층 구조 결정
  3. 주제 정의 - 관련 객체를 논리적 그룹으로 조직화
  4. 속성 정의 - 각 객체에 대한 데이터 속성 정의
  5. 관계 정의 - 객체 간 연결 설명
  6. 서비스 정의 - 객체가 수행할 수 있는 작업 지정

1. 객체 식별

객체 식별은 객체로 표현될 시스템에서 중요한 엔티티를 찾는 과정입니다. 객체는 데이터(속성)와 동작(서비스)을 하나의 단위로 결합합니다.

객체 식별 방법:

  • 주요 명사를 찾기 위해 요구사항 문서 분석
  • CRC(클래스-책임-협력) 카드 사용
  • 브레인스토밍 세션 진행 및 도메인 전문가 인터뷰
객체 표기법 

┌─────────────────┐
│     Member      │ ← Class name
├─────────────────┤
│ - memberId      │ 
│ - name          │ ← Attribute area
│ - contact       │
├─────────────────┤
│ + borrowBook()  │ ← Service (behavior) area
│ + returnBook()  │
└─────────────────┘

도서관 시스템 예시에서 다음과 같은 주요 객체를 식별할 수 있습니다:

  • 회원: 도서관 서비스를 이용하는 사람
  • 책: 대출할 수 있는 자료
  • 사서: 도서관을 관리하는 직원
  • 대출: 회원이 책을 빌린 기록

2. 구조 식별

구조 식별은 객체 간의 관계, 특히 일반화/특수화(상속) 및 전체/부분(집합) 관계를 결정하는 것을 포함합니다.

주요 관계 유형:

  • 상속(is-a): 부모-자식 클래스 관계 식별
  • 집합(has-a): 한 객체가 다른 객체를 포함하는 경우 식별
classDiagram class LibraryItem { +itemId: String +title: String +status: Status +checkOut() +checkIn() } class Book { +author: String +isbn: String } class Periodical { +issueNo: Integer +pubDate: Date } LibraryItem <|-- Book LibraryItem <|-- Periodical

LibraryItem을 부모 클래스로 보여주는 상속 관계 다이어그램

classDiagram class Library { +name: String +location: String } class Department { +name: String +floor: Integer } Library "1" o-- "*" Department : contains

도서관과, 부서 간의 집합 관계(전체-부분 관계)

3. 주제 정의

주제 정의는 관련 객체를 논리적 그룹이나 하위 시스템으로 구성하여 복잡한 시스템을 더 관리하기 쉽게 만드는 것을 포함합니다.

그룹화 기준:

  • 기능적 연관성
  • 도메인 영역
  • 비즈니스 프로세스 정렬

주제는 관련 객체를 논리적 하위 시스템으로 그룹화합니다

주제 표기법

고객 관리 시스템템 ← Another subject

│  └─────────┘  └───────────┘  └──────────┘   │

4. 속성 정의

속성은 객체의 상태와 특성을 나타내는 데이터 항목입니다. 각 객체는 여러 속성을 가질 수 있습니다.

속성 유형:

  • 기본 키: 객체를 고유하게 식별
  • 필수 속성: 값이 있어야 함
  • 선택적 속성: 값이 있거나 없을 수 있음
  • 파생 속성: 다른 속성에서 계산할 수 있음
속성 표기법 

┌───────────────────────────────┐
│           Book                │
├───────────────────────────────┤
│ - isbn: String {PK}           │ ← Primary key attribute
│ - title: String {not null}    │ ← Required attribute
│ - author: String {not null}   │
│ - publisher: String           │ ← Regular attribute
│ - publicationYear: Integer    │
│ - price: Decimal              │
│ - availableCount: Integer     │ ← Derived attribute
├───────────────────────────────┤
│ + getAvailability(): Boolean  │
└───────────────────────────────┘

5. 관계 정의

관계는 객체 간의 연결을 나타냅니다. 코드-유돈 방법론은 이러한 관계를 시각화하기 위해 E-R 다이어그램을 사용합니다.

관계 유형:

  • 일대일(1:1): 한 객체가 정확히 하나의 다른 객체와 관련됨
  • 일대다(1:*): 한 객체가 여러 객체와 관련됨
  • 다대다(*:*): 여러 객체가 여러 다른 객체와 관련됨
classDiagram class Member { +memberId: String +name: String +borrowBook() } class Loan { +loanId: String +issueDate: Date +dueDate: Date +renew() } class Book { +isbn: String +title: String +checkStatus() } Member "1" --> "*" Loan : borrows Loan "*" --> "1" Book : references

회원, 대출, 책 간의 연결을 보여주는 관계 다이어그램

전문가 팁

관계를 정의할 때 모델을 더 명확하게 만들기 위해 항상 카디널리티(1:1, 1:*, *:*)와 방향을 지정하세요. 또한 "대여" 또는 "관리"와 같이 관계의 목적을 설명하는 이름을 각 관계에 부여하는 것을 고려하세요.

6. 서비스 정의

서비스는 객체가 수행할 수 있는 동작이나 작업을 정의합니다. 이들은 객체의 기능적 측면을 나타냅니다.

서비스 요소:

  • 입력 매개변수
  • 반환 값
  • 가시성(공개, 비공개)
  • 서비스 상호작용
서비스 표기법 

┌───────────────────────────────────────────────────┐
│                   Member                          │
├───────────────────────────────────────────────────┤
│- memberId: String                                 │
│- name: String                                     │
│- email: String                                    │
├───────────────────────────────────────────────────┤
│+ borrowBook(bookId: String): Loan                 │ ← Method with parameters
│+ returnBook(loanId: String): Boolean              │
│+ searchBooks(keyword: String): List               │ ← Return type
│+ updateProfile(name: String, email: String): void │
│- validateMemberStatus(): Boolean                  │ ← Private method
└───────────────────────────────────────────────────┘

실제 예제: 도서관 관리 시스템

도서관 관리 시스템에 적용된 코드-유돈 방법론의 완전한 예를 살펴보겠습니다.

classDiagram class Person { +id: String +name: String +contact: String +register() +updateInfo() } class Member { +memberId: String +status: Status +borrowBook() +returnBook() } class Librarian { +staffId: String +department: String +addBook() +issueCard() } class Loan { +loanId: String +issueDate: Date +dueDate: Date +returnDate: Date +create() +renew() +complete() } class LibraryItem { +itemId: String +title: String +status: Status +checkOut() +checkIn() } class Book { +isbn: String +author: String +reserve() } class Periodical { +issueNo: Integer +publicationDate: Date +archive() } class Reservation { +reserveId: String +reserveDate: Date +status: Status +create() +cancel() +convert() } Person <|-- Member Person <|-- Librarian Member "1" --> "*" Loan : borrows Loan "*" --> "1" LibraryItem : references LibraryItem <|-- Book LibraryItem <|-- Periodical Member "1" --> "*" Reservation : makes Book "1" --> "*" Reservation : has Librarian --> LibraryItem : manages

코드-유돈 방법론을 사용한 도서관 관리 시스템에 대한 완전한 E-R 다이어그램

이 다이어그램은 코드-유돈 방법론의 모든 여섯 단계가 어떻게 결합되는지 보여줍니다:

  1. 객체 식별: 사람, 회원, 사서, 대출, 도서관 항목 등
  2. 구조 식별: 사람→회원/사서, 도서관항목→책/정기물
  3. 주제 정의: 전체 시스템은 "도서관 관리 시스템"
  4. 속성 정의: 각 객체는 자체 속성(아이디, 이름 등)을 가짐
  5. 관계 정의: 대여, 참조, 관리 등
  6. 서비스 정의: 책대여(), 대출(), 등

장점과 한계

장점 한계
직관적인 시각적 표현 복잡한 동적 동작에 대한 제한된 지원
객체 지향 분석에 대한 체계적 접근 구현 세부사항을 다루지 않음
데이터와 동작 모두에 중점 매우 큰 시스템에는 잘 확장되지 않을 수 있음
설계 단계로의 자연스러운 전환 현대 개발에서 UML보다 덜 인기 있음

유산과 현대적 영향

코드-유돈 방법론은 대부분 통합 모델링 언어(UML)로 대체되었지만, 객체 지향 분석에 대한 그 개념과 접근 방식은 이 분야에 상당한 영향을 미쳤습니다. 그 원칙 중 많은 부분이 UML 개발에 영향을 미쳤습니다.

지속적인 기여

  • 시스템의 시각적 모델링 강조
  • 객체에서 데이터와 동작의 통합
  • 시스템 분석에 대한 구조화된 접근
  • 재사용성과 모듈성에 중점
